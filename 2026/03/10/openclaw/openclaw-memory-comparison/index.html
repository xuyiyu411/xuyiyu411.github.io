方案概览

方案 类型 状态 核心特点
SQLite + FTS5 原生内置 生产就绪 文件优先，Markdown 源码为真理
QMD 原生扩展 实验性 BM25 + 向量混合检索，完全离线
LanceDB 第三方插件 社区维护 高性能向量数据库，混合检索

原生方案：SQLite + FTS5

相关链接

  • 官方文档：https://docs.openclaw.ai/concepts/memory
  • GitHub：https://github.com/openclaw/openclaw

架构原理

OpenClaw 原生记忆系统采用文件优先的设计理念：

  • 存储层：Markdown 文件 (memory/YYYY-MM-DD.md, MEMORY.md)
  • 索引层：SQLite + FTS5 全文检索 + 可选向量索引
  • 检索层：语义搜索 (向量) + 关键词搜索 (FTS5) 混合

环境依赖

项目 要求
Node.js 22+
SQLite 内置 (Node.js 原生支持)
嵌入模型 可选：OpenAI/Gemini/Voyage/Mistral API 或本地 Ollama

资源需求

  • 磁盘：极低，仅 SQLite 数据库 + Markdown 文件
  • 内存：低，FTS5 查询内存占用小
  • CPU：低，无复杂计算
  • 网络：可选 (仅远程嵌入需要)

使用成本

  • 基础设施：零成本，纯本地运行
  • API 费用：如使用远程嵌入，按 token 计费；本地 Ollama 零费用
  • 维护成本：低，自动索引

安全性

  • 数据隐私：⭐⭐⭐⭐⭐ 完全本地存储
  • 数据可控：⭐⭐⭐⭐⭐ Markdown 可直接编辑审计
  • 依赖风险：⭐⭐⭐⭐⭐ 无外部依赖

优缺点

优点：

  • 开箱即用，零配置
  • 透明可审计，Markdown 为人类可读格式
  • 自动选择嵌入提供商
  • 支持向量检索降级到纯 FTS5

缺点：

  • 大规模数据时向量检索性能受限
  • 无自动记忆提取，依赖 Agent 主动写入
  • 缺乏记忆整合与去重机制

QMD 后端

相关链接

  • 官方文档：https://docs.openclaw.ai/concepts/memory
  • QMD 仓库：https://github.com/tobi/qmd

架构原理

QMD (Query Markdown Documents) 是一个实验性的替代索引后端，替换内置 SQLite 管理器：

  • 检索算法：BM25 + 向量 + 重排序 (Reciprocal Rank Fusion)
  • 嵌入模型：完全本地，通过 node-llama-cpp 自动下载 GGUF 模型
  • 运行方式：作为独立 sidecar 进程运行

环境依赖

项目 要求
Node.js 22+
Bun 必须
SQLite 支持扩展的版本 (brew install sqlite)
QMD CLI 需单独安装 (bun install -g qmd)
OS macOS / Linux (Windows 需 WSL2)

资源需求

  • 磁盘：中等，需存储 GGUF 嵌入模型 (~几百 MB)
  • 内存：中等，本地模型加载需额外内存
  • CPU：中等，首次搜索需加载模型
  • 网络：仅首次下载模型需要

使用成本

  • 基础设施：零成本
  • API 费用：零费用，完全本地
  • 维护成本：中等，需手动安装和维护 QMD

安全性

  • 数据隐私：⭐⭐⭐⭐⭐ 完全离线
  • 数据可控：⭐⭐⭐⭐⭐ 本地模型，无数据外传
  • 依赖风险：⭐⭐⭐ 依赖 Bun 和 QMD 生态

优缺点

优点：

  • 检索质量显著优于纯 FTS5
  • 完全离线，无需 API Key
  • 混合排序 (BM25 + 向量 + 重排序)
  • 自动回退到内置管理器

缺点：

  • 安装复杂，需 Bun + QMD CLI
  • 仅支持 macOS/Linux
  • 首次搜索慢 (模型下载)
  • 实验性，API 可能变动

LanceDB 方案

相关链接

  • LanceDB 官网：https://lancedb.com/
  • LanceDB GitHub：https://github.com/lancedb/lancedb
  • OpenClaw 插件：https://github.com/win4r/memory-lancedb-pro

架构原理

LanceDB 是一个开源向量数据库，通过社区插件与 OpenClaw 集成：

  • 存储格式：Lance 列式格式 (Apache Arrow 兼容)
  • 检索能力：ANN 向量搜索 + BM25 混合
  • 架构：嵌入式/无服务器，无需独立服务

环境依赖

项目 要求
Node.js 22+
LanceDB 通过 npm 安装
Rust 编译依赖 (可选预构建二进制)

资源需求

  • 磁盘：中等，列式存储效率高
  • 内存：中等，向量索引常驻内存
  • CPU：中等，ANN 搜索需计算
  • 网络：无 (纯本地)

使用成本

  • 基础设施：零成本
  • API 费用：零费用
  • 维护成本：低，通过插件管理

安全性

  • 数据隐私：⭐⭐⭐⭐⭐ 完全本地
  • 数据可控：⭐⭐⭐⭐⭐ 文件格式开放
  • 依赖风险：⭐⭐⭐⭐ 社区插件维护

优缺点

优点：

  • 高性能向量检索
  • 支持多模态数据
  • 版本控制 (时间旅行)
  • 生产级可靠性 (Uber 等使用)

缺点：

  • 社区插件，非官方支持
  • 配置相对复杂
  • 需要理解向量数据库概念

Mem0

相关链接

  • 官方网站：https://mem0.ai/
  • GitHub：https://github.com/mem0ai/mem0

架构原理

Mem0 是一个智能记忆层，专注于自动提取和整合：

  • 记忆类型：episodic (情景) + semantic (语义) + procedural (程序性)
  • 提取机制：LLM 自动从对话中提取事实
  • 整合策略：去重、冲突解决、置信度更新

Mem0 本质上是一个 RAG 系统，通过文档分块、嵌入和检索实现记忆功能。

环境依赖

项目 要求
Python 3.9+
数据库 PostgreSQL / SQLite / Redis (可选)
LLM API OpenAI / Anthropic / 本地模型

资源需求

  • 磁盘：取决于存储后端
  • 内存：中等，需运行 Mem0 服务
  • CPU：高，频繁调用 LLM 进行提取
  • 网络：必须，依赖 LLM API

使用成本

  • 基础设施：自托管免费，云服务付费
  • API 费用：高，每次提取都需 LLM 调用
  • 维护成本：中等，需管理 Mem0 服务

安全性

  • 数据隐私：⭐⭐⭐ 依赖外部 LLM API
  • 数据可控：⭐⭐⭐ 数据存储为嵌入，不可直接审计
  • 依赖风险：⭐⭐ 依赖第三方服务和库

优缺点

优点：

  • 自动记忆提取，无需 Agent 主动写入
  • 智能去重和冲突解决
  • 支持多用户/多 Agent 隔离
  • 成熟的商业支持 (YC 背书)

缺点：

  • 成本高，频繁 LLM 调用
  • 依赖外部服务
  • 与 OpenClaw 集成需适配层
  • 记忆存储为嵌入，不可直接审计

MemU

相关链接

  • GitHub：https://github.com/NevaMind-AI/memU
  • 官网：https://memu.pro/

架构原理

MemU 是一个开源记忆框架，采用类文件系统的层级结构：

  • 组织方式：文件夹 (类别) → 文件 (记忆项) → 符号链接 (关联)
  • 核心特性：主动记忆 (Proactive Memory) + 自动组织
  • 检索机制：不依赖嵌入，基于结构和关联

MemU 与 Mem0 的本质区别：MemU 是纯记忆层，不是 RAG 系统。

环境依赖

项目 要求
Python 3.9+
存储 本地文件系统
LLM 可选，用于增强提取

资源需求

  • 磁盘：低，纯文本存储
  • 内存：低
  • CPU：低
  • 网络：无 (纯本地)

使用成本

  • 基础设施：零成本
  • API 费用：零费用 (可选 LLM 增强)
  • 维护成本：中等，需理解其文件系统隐喻

安全性

  • 数据隐私：⭐⭐⭐⭐⭐ 完全本地
  • 数据可控：⭐⭐⭐⭐⭐ 纯文本，完全透明
  • 依赖风险：⭐⭐⭐⭐ 开源项目，社区驱动

优缺点

优点：

  • 创新的层级组织方式
  • 不依赖嵌入，避免向量搜索的不确定性
  • 支持主动记忆 (Agent 可自主整理)
  • 完全开源，无商业锁定
  • Token 消耗降低 70%+

缺点：

  • 概念新颖，学习曲线较陡
  • 与 OpenClaw 集成需开发适配
  • 社区相对较小

MemOS

相关链接

  • GitHub：https://github.com/MemTensor/MemOS
  • 官网：https://memos.openmem.net/
  • 论文：https://arxiv.org/abs/2507.03724

架构原理

MemOS 是一个内存操作系统概念框架，来自学术研究：

  • 核心理念：将内存视为可系统管理的资源 (与 CPU、存储并列)
  • 三层架构：
    • Interface Layer (API 与应用接口层)
    • Operation Layer (记忆调度与管理层)
    • Infrastructure Layer (存储与基础设施层)
  • MemCube：统一记忆容器，支持明文、激活 (KV Cache)、参数化 (LoRA) 三种记忆类型

环境依赖

项目 要求
Python 3.10+
存储 可插拔 (SQLite / LanceDB / 云存储)
基础设施 较复杂

资源需求

  • 磁盘：取决于配置
  • 内存：中高，需运行完整服务栈
  • CPU：中高
  • 网络：取决于存储后端

使用成本

  • 基础设施：自托管成本
  • API 费用：取决于 LLM 使用
  • 维护成本：高，概念复杂

安全性

  • 数据隐私：⭐⭐⭐⭐ 取决于配置
  • 数据可控：⭐⭐⭐⭐ 开源，但架构复杂
  • 依赖风险：⭐⭐ 研究阶段，API 不稳定

优缺点

优点：

  • 学术前沿，理论完整
  • 统一多种记忆类型 (明文 + 激活 + 参数化)
  • 支持企业级治理 (合规、审计、生命周期)
  • Token 消耗降低 72%+
  • 多 Agent 记忆共享

缺点：

  • 研究阶段，生产就绪度低
  • 架构复杂，部署困难
  • 与 OpenClaw 集成工作量巨大

场景推荐

迁移路径建议

  1. 阶段 1：使用原生 SQLite+FTS5 建立基础工作流
  2. 阶段 2：当检索质量成为瓶颈时，迁移到 QMD 或 LanceDB
  3. 阶段 3：需要自动记忆管理时，评估 Mem0 或自建提取层
  4. 阶段 4：Token 成本或企业治理需求出现时，考虑 MemU 或 MemOS

延伸阅读

核心概念

  • RAG vs 纯记忆：Mem0 是 RAG 系统 (文档分块 + 嵌入检索)，MemU 是纯记忆层 (结构化存储)
  • 混合检索：QMD 结合 BM25 (关键词) + 向量 (语义) + 重排序，检索质量最优
  • Token 优化：MemU 和 MemOS 通过精准记忆检索，避免加载完整对话历史，显著降低 Token 消耗

结论

OpenClaw 的记忆架构设计遵循渐进增强原则：

  • 原生 SQLite+FTS5 足以满足 80% 用户的需求
  • QMD 是追求离线高质量检索的最佳选择
  • LanceDB 适合需要生产级向量数据库的场景
  • Mem0 适合追求自动化且接受 API 成本的场景
  • MemU 适合 Token 成本敏感且偏好透明存储的场景
  • MemOS 代表未来方向，适合企业级多 Agent 治理

核心建议：从简单开始，按需升级。OpenClaw 的文件优先设计确保了无论选择哪种后端，Markdown 源码始终是真理，可随时迁移或重建索引。
