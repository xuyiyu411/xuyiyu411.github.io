OpenClaw 闲谈：2026.3.2 版本权限管控功能

引言

新安装的 OpenClaw 2026.3.2 版本，你会发现它既不能帮你操作文件，也不能帮你安装skill。这难道是因为模型降智了吗？其实不是，而是因为 OpenClaw 2026.3.2 版本引入了更加精细化的权限管控机制，旨在提升系统安全性与可控性。本文档将详细介绍该版本新增的权限配置选项、使用场景及最佳实践。

一、核心权限架构

1.1 权限管控层级

OpenClaw 的权限管控采用三层架构：

层级 作用范围 优先级
全局默认 (agents.defaults) 所有 Agent 的默认行为 最低
Agent 级别 (agents.list[].tools) 单个 Agent 的工具权限 中等
沙箱级别 (tools.sandbox.tools) 沙箱环境中的工具限制 最高（沙箱内）

2.1 基础配置结构

在 ~/.openclaw/openclaw.json 中，tools 配置段控制所有工具相关的权限：

2.2 权限配置字段说明

tools.profile - 权限模板

OpenClaw 提供预设的权限模板，简化配置：

模板名称 包含的工具组 适用场景
minimal 仅 session_status 最小权限
coding group:fs, group:runtime, group:sessions, group:memory, image 开发辅助（默认）
messaging group:messaging, sessions_list, sessions_history, sessions_send, session_status 消息通信
full 无限制（等同于未设置） 完全开放

默认值: coding（本地 onboarding 时如果未设置则默认为 "coding"，现有显式配置会被保留）

工具组说明:

工具组 包含的工具
group:fs read, write, edit, apply_patch
group:runtime exec, process, bash
group:sessions sessions_list, sessions_history, sessions_send, sessions_spawn, session_status
group:memory memory_search, memory_get
group:messaging message
group:web web_search, web_fetch
group:ui browser, canvas
group:automation cron, gateway
group:nodes nodes
group:openclaw 所有内置工具（不包括 provider 插件）

tools.allow / tools.deny - 显式授权/禁止

  • 白名单模式：若设置 allow，则仅允许列表中的工具可用
  • 黑名单模式：若设置 deny，则除列表外的所有默认可用工具都可用
  • 混合模式：同时设置时，deny 优先级高于 allow
  • 大小写不敏感：工具名称不区分大小写
  • 通配符支持：支持 * 通配符（如 "*" 表示所有工具）
  • 沙箱无关：即使 Docker 沙箱关闭，此策略仍然生效

注意：如果 tools.allow 仅引用未知或未加载的插件工具名称，OpenClaw 会记录警告并忽略该允许列表，以确保核心工具仍然可用

完整工具清单：

工具类别 工具名称 风险等级 说明
文件操作 read 低 读取文件内容
文件操作 write 中 创建/覆盖文件
文件操作 edit 中 精确编辑文件
代码执行 exec 高 执行 shell 命令
代码执行 process 高 管理后台进程
网络浏览 browser 中 浏览器自动化
网络浏览 web_search 低 网页搜索
网络浏览 web_fetch 低 获取网页内容
图形界面 canvas 中 画布操作
设备控制 nodes 高 节点设备控制
消息通信 message 中 发送消息
会话管理 sessions_spawn 高 创建子会话
系统管理 gateway 极高 网关控制
媒体处理 image 低 分析图片（需配置 imageModel）
文档处理 pdf 低 分析 PDF 文档
补丁应用 apply_patch 中 应用结构化补丁（多文件编辑）

tools.elevated - 特权操作控制

特权工具需要额外授权才能使用：

特权工具包括：

  • exec（shell 命令执行）
  • process（进程管理）
  • gateway（网关重启/停止）
  • nodes（远程设备控制）
  • sessions_spawn（子代理创建）

注意：

  • Agent 级别的 tools.elevated 只能进一步限制，不能放宽全局设置
  • /elevated on|off|ask|full 命令按会话存储状态；内联指令仅应用于单条消息
  • 特权 exec 在主机上运行，绕过沙箱隔离

tools.exec - 执行工具配置

tools.byProvider - 按提供商限制

可针对特定模型提供商进一步限制工具权限。生效顺序：基础 profile → 提供商 profile → allow/deny。

tools.sessions - 会话工具可见性

控制会话工具（sessions_list, sessions_history, sessions_send）可访问的会话范围：

可见性选项：

  • self: 仅当前会话
  • tree: 当前会话 + 由当前会话派生的会话（子代理）- 默认
  • agent: 属于当前 Agent ID 的任何会话（如果使用 per-sender 会话，可能包含其他用户）
  • all: 任何会话。跨 Agent 定位仍需 tools.agentToAgent

沙箱限制： 当当前会话处于沙箱中且 agents.defaults.sandbox.sessionToolsVisibility="spawned" 时，即使设置 tools.sessions.visibility="all"，可见性也会被强制限制为 tree

tools.agentToAgent - 跨 Agent 通信

控制是否允许 Agent 间通信：

  • 需要启用此选项才能使用 sessions_send 向其他 Agent 的会话发送消息
  • 即使 tools.sessions.visibility 设置为 all，跨 Agent 定位仍需启用此选项

tools.web - Web 工具配置

tools.media - 媒体处理配置

配置入站媒体理解（图片/音频/视频）：

媒体模型类型：

  • Provider 条目 (type: "provider" 或省略): 使用 provider, model, profile 字段
  • CLI 条目 (type: "cli"): 使用 command, args 字段，支持模板如 {{MediaPath}}, {{Prompt}}, {{MaxChars}}

tools.loopDetection - 循环检测配置

工具循环安全检查默认禁用，设置 enabled: true 激活检测：

配置说明：

  • historySize: 循环分析保留的最大工具调用历史数
  • warningThreshold: 重复无进展模式警告阈值
  • criticalThreshold: 阻止关键循环的更高阈值
  • globalCircuitBreakerThreshold: 任何无进展运行的硬停止阈值
  • 可在 Agent 级别通过 agents.list[].tools.loopDetection 覆盖全局设置
  • 验证规则：warningThreshold < criticalThreshold < globalCircuitBreakerThreshold

3.2 权限继承规则

Agent 级别的 tools 配置会合并覆盖全局默认值：

继承规则：

  • 若 Agent 未定义 tools 字段，则继承 agents.defaults.tools
  • Agent 级别显式设置的字段会覆盖全局默认值
  • tools.elevated 在 Agent 级别只能进一步限制，不能放宽全局设置

子代理控制：

  • agents.list[].subagents.allowAgents: 允许 spawn 的 Agent ID 列表（["*"] 表示允许任何 Agent，默认仅允许相同 Agent）
  • tools.subagents.tools.allow / tools.subagents.tools.deny: 控制子代理可用的工具
  • 沙箱继承保护：如果请求会话处于沙箱中，sessions_spawn 会拒绝那些将以非沙箱方式运行的目标

四、沙箱环境权限隔离

4.1 沙箱工具限制

当启用 Docker 沙箱时，可在沙箱内进一步限制可用工具：

Workspace access 说明：

  • none: 在 ~/.openclaw/sandboxes 下创建独立的沙箱工作区
  • ro: 沙箱工作区挂载到 /workspace，Agent 工作区以只读方式挂载到 /agent
  • rw: Agent 工作区以读写方式挂载到 /workspace

Scope 说明：

  • session: 每个会话独立的容器 + 工作区
  • agent: 每个 Agent 一个容器 + 工作区（默认）
  • shared: 共享容器和工作区（无跨会话隔离）

4.2 沙箱权限优先级

沙箱内的工具权限受双重限制：

  1. 外层限制：Agent 的 tools.allow/deny 决定能否进入沙箱
  2. 内层限制：tools.sandbox.tools 决定在沙箱内能做什么

七、故障排查

7.1 常见问题

Q: 为什么工具调用被拒绝了？

检查以下配置点：

  1. Agent 级别：确认 agents.list[].tools.allow 包含该工具
  2. 特权检查：高风险工具需要在 elevated.allowFrom 中授权
  3. 沙箱限制：如果在沙箱中运行，检查 tools.sandbox.tools

Q: 如何查看当前生效的权限？

使用 /debug 命令（需先启用）查看当前会话的权限状态。

Q: 权限修改后何时生效？

  • 大多数配置支持热重载（hot reload），修改后自动生效，无需重启 Gateway
  • 仅以下类别需要重启：
    • Gateway 服务器配置（gateway.*）：端口、绑定地址、认证、Tailscale、TLS、HTTP 等
    • 基础设施配置（discovery、Tailscale 等）
  • 工具权限、Agent 配置、频道配置等均支持热重载

八、版本变更说明

2026.3.2 版本更新要点

  1. **新增 tools.profile**：预设权限模板，简化配置
  2. **增强 elevated.allowFrom**：支持更细粒度的渠道特定授权
  3. 改进沙箱权限隔离：tools.sandbox.tools 独立于主权限体系
  4. 优化权限继承逻辑：Agent 级别配置现在完全覆盖全局默认值

从旧版本迁移

若从 2026.2.x 升级，请注意：

  • 原有 tools 配置仍然兼容
  • 建议逐步迁移到新的 profile 模板语法
  • elevated.enabled 默认为 false，如需特权操作请显式开启
